# 초등 영단어 스펠링 게임 개발 프롬프트 (바이브 코딩용)

아래 사양대로 웹 기반 실시간 영단어 학습 게임을 처음부터 끝까지 구현해줘.
프론트엔드/백엔드 전체 코드, 폴더 구조, 실행 방법(README)까지 포함해서 만들어줘.

---

## 1. 프로젝트 개요

- 대상: 초등학교 3~6학년
- 목적: 영단어를 빠르고 정확하게 스펠링으로 완성하는 실시간 퀴즈 게임
- 형태: 브라우저 기반 웹앱 (PC/태블릿 반응형)
- 모드: 싱글플레이 / 멀티플레이 (한 방 최대 약 30명 동시 접속)
- 참고 포맷: 카훗(Kahoot) 류의 실시간 동시 진행 방식이지만, 정답은 객관식이 아니라 "알파벳 버튼을 순서대로 클릭해서 단어를 완성"하는 방식

---

## 2. 핵심 게임 플레이 규칙 (정확히 구현할 것)

한 문제(라운드)는 다음 순서로 진행돼:

1. **단어 노출 단계**: 화면에 영어 단어 원문이 설정된 시간(기본 3초, 조정 가능) 동안 표시됨
2. **시간 종료 후**: 영어 단어는 사라지고, 해당 단어의 **한글 뜻만** 화면에 표시됨
3. 한글 뜻 아래에 **정답 스펠링에 해당하는 알파벳들이 무작위 순서로 섞여서 버튼 형태**로 나타남
   - 기본값: 정답 글자 수만큼의 알파벳만 제공 (예: apple → a, p, p, l, e를 랜덤 배치)
   - 옵션(선택 구현): 오답 알파벳 1~2개를 섞어 난이도를 높이는 모드도 추가 가능하게 설계해줘 (난이도가 "고급/최고급"일 때 활성화되는 옵션으로 제안)
4. 학생은 알파벳 버튼을 순서대로 클릭해서 단어를 완성함 (클릭한 순서대로 상단에 조합된 글자가 쌓여 보이고, 잘못 눌렀을 경우 되돌리기/지우기 버튼 제공)
5. 제출은 자동(글자 수를 다 채우면 자동 채점) 또는 "제출" 버튼 클릭으로 처리
6. **정답을 빨리 제출할수록 높은 점수** 획득 (점수 공식은 6번 항목 참고)
7. 모르면 **"패스" 버튼**으로 넘어갈 수 있음 (0점 처리)
8. **방에 참가한 모든 학생이 정답 제출 또는 패스를 완료하면 자동으로 다음 문제로 전환**
   - 단, 일부 학생이 응답하지 않고 멈춰있는 경우를 대비해 **최대 대기시간(문제 풀이 제한시간)**이 지나면 강제로 다음 문제로 넘어가도록 타임아웃 로직도 함께 구현
9. 설정된 문제를 모두 마치면 **최종 랭킹 화면**으로 이동

싱글플레이도 동일한 흐름으로 진행하되, "전원 제출 대기" 로직 없이 본인이 답을 내면 바로 다음 문제로 넘어감.

## 3. 난이도 체계

5단계: **연습 / 기본 / 중급 / 고급 / 최고급**

- 단어 데이터는 난이도별로 분리해서 JSON 파일로 관리 (`words_practice.json`, `words_basic.json`, `words_intermediate.json`, `words_advanced.json`, `words_expert.json` 등)
- 학년 매핑 가이드(초기 데이터 제작 기준으로 참고):
  - 연습: 워밍업용, 3~4글자 쉬운 단어 (전 학년 공용)
  - 기본: 3~4학년 수준 단어
  - 중급: 4~5학년 수준 단어
  - 고급: 5~6학년 수준 단어
  - 최고급: 6학년 심화/도전 단어, 단어 길이 김
- 난이도가 올라갈수록 기본값으로 단어 노출 시간은 짧게, 단어 길이는 길게, 풀이 제한시간은 짧게 설정되도록 초기값을 제안하되, **최종적으로는 아래 4번 항목처럼 사용자가 직접 시간/문제 수를 조정할 수 있어야 함**
- 단어 데이터는 추후 교사가 쉽게 추가/수정할 수 있도록 `{ id, word, meaning, difficulty }` 형태의 간단한 JSON 구조로 설계

## 4. 첫 화면(로비) 설계

첫 화면에 아래 요소들이 모두 포함되어야 함:

1. **모드 선택**: 싱글플레이 / 멀티플레이 버튼
2. **닉네임 입력**: 필수 입력, 멀티플레이의 경우 같은 방 안에서 닉네임 중복 불가 (중복 시 안내 메시지 후 재입력 요구)
3. **게임 옵션 설정** (싱글은 본인이, 멀티는 방장만 설정 가능)
   - 난이도 선택 (연습/기본/중급/고급/최고급 중 택1)
   - 단어 노출 시간 설정 (예: 1~10초 슬라이더, 기본 3초)
   - 문제 풀이 제한시간 설정 (예: 5~30초 슬라이더, 기본 10초)
   - 문제 개수 설정 (예: 5~30문제, 기본 10문제)
4. **멀티플레이 전용 UI**
   - "방 만들기": 6자리 랜덤 방 코드 발급 후 대기실로 이동
   - "방 참가하기": 방 코드 입력 후 입장 (방이 없거나 인원이 꽉 찬 경우 에러 메시지)
   - 대기실에서 현재 참가자 목록(닉네임)과 인원수(예: 12/30) 실시간 표시
   - 방장만 "게임 시작" 버튼 활성화
5. **점수/랭킹 시스템**
   - "이번 게임 랭킹"(라운드 종료 후 표시)과 별개로, 원한다면 "역대 랭킹" 페이지도 추가 제안 (로컬 저장 또는 서버 저장, 선택 구현)

---

## 5. 게임 진행 화면

- 현재 문제 번호 / 전체 문제 수 표시 (예: 3 / 10)
- 진행 상태에 따라 타이머 UI 전환: 단어 노출 타이머 → 풀이 제한시간 타이머
- 멀티플레이 시:
  - 다른 참가자들의 제출 현황을 "몇 명 제출완료 / 전체 인원" 형태로만 표시 (누가 어떤 답을 썼는지는 절대 노출하지 않음 — 부정행위 방지)
  - 전원 제출 또는 타임아웃 시 자동으로 다음 문제로 전환되는 것을 화면 전환 애니메이션으로 명확히 보여줄 것
- 정답/오답 즉시 피드백 (색상, 애니메이션, 효과음 등)
- 알파벳 버튼 UI는 터치 친화적으로 크게 (초등학생 대상, 태블릿 사용 고려)

---

## 6. 점수 시스템

점수 계산 공식 예시 (그대로 구현하거나 유사하게 조정 가능):

```
기본점수 = 100점 (정답 시)
속도가산점 = 남은시간(초) / 제한시간(초) * 100
최종점수 = 기본점수 + 속도가산점 (최대 200점)

오답 제출 시 = 0점
패스 시 = 0점
```

- 난이도가 높을수록(고급/최고급) 문제당 배점에 가중치(예: x1.2, x1.5)를 곱하는 옵션도 제안
- 게임 진행 중 실시간 누적 점수를 서버에서 관리하고, 각 문제 종료 시 참가자에게 브로드캐스트

---

## 7. 랭킹 화면

- 모든 문제가 끝나면 자동으로 랭킹 화면 표시
- 최종 순위, 닉네임, 총점, 정답률(정답 수/전체 문제 수) 표시
- 상위 3명은 시상대(1위/2위/3위) 형태의 UI로 강조
- 멀티플레이는 전체 참가자 랭킹을 실시간으로 서버에서 계산해 방 전체에 브로드캐스트

---

## 8. 멀티플레이 방(Room) 관리 — 자원 낭비 방지 (필수 요구사항)

서버에서 방(room)을 다음과 같이 관리해줘:

1. 방 생성 시 6자리 고유 코드 발급, 최대 인원 30명으로 제한 (초과 시 입장 거부)
2. 방 상태를 `대기중 / 게임중 / 종료`로 관리
3. **자동 삭제(정리) 조건**을 반드시 구현:
   - 방에 참가자가 0명이 되면 일정 유예시간(예: 30초) 후 자동 삭제
   - 대기중 상태로 일정 시간(예: 10분) 이상 아무 활동이 없으면 자동 삭제
   - 게임 종료(랭킹 화면 도달) 후 일정 시간(예: 5분) 이상 방치되면 자동 삭제
   - 방장이 연결 끊김(disconnect) 상태로 유예시간(예: 30초~1분) 동안 재접속하지 않으면, 다른 참가자에게 방장 권한을 위임하거나 방을 종료
4. 서버에 **주기적 클린업 작업**(예: 1분 간격 interval 또는 cron)을 두어 조건에 해당하는 방들을 스캔하고 정리하도록 구현
5. 각 방 객체에 `lastActivityAt` 타임스탬프를 두고, 참가/제출/메시지 등 활동이 있을 때마다 갱신해서 정리 로직의 판단 기준으로 사용

## 9. 동시 접속자 처리 시 고려사항 (중요)

여러 개의 방(각 방 최대 30명)이 동시에 열릴 수 있는 상황을 가정하고 다음을 반영해줘:

- 실시간 통신은 **Socket.io**를 사용하고, 방마다 Socket.io room으로 격리해서 불필요한 브로드캐스트가 다른 방에 전달되지 않도록 할 것
- 방/게임 상태는 방 단위로 서버 메모리에 분리 저장 (`Map<roomId, RoomState>` 구조), 방 하나의 상태 변경이 다른 방에 영향을 주지 않도록 설계
- 클릭 스팸이나 중복 제출을 막기 위한 서버 측 입력 검증(한 문제당 한 번만 제출 허용 등) 추가
- 새로고침이나 순간적인 연결 끊김에도 게임 진행 상태를 잃지 않도록 **세션 토큰(또는 소켓 재연결 시 방 코드+닉네임으로 재입장) 처리** 구현
- 프론트엔드에서는 다른 참가자의 제출 현황처럼 자주 갱신되는 값이 전체 UI를 불필요하게 리렌더링하지 않도록 상태를 분리 관리 (예: 제출 현황 카운터만 별도 컴포넌트로 분리)
- 초기 규모(동시 몇 개 방 x 30명 수준)는 단일 Node.js 서버 + 메모리 상태로 충분하지만, 이후 트래픽이 늘어날 경우를 대비해 Socket.io의 Redis adapter로 손쉽게 수평 확장할 수 있는 구조로 설계해줘 (지금 당장 Redis를 붙이지 않아도 되지만, 상태 관리 로직을 나중에 Redis로 옮기기 쉽게 모듈화할 것)

## 10. 기술 스택 제안

- **프론트엔드**: React + Vite, 실시간 통신은 `socket.io-client`, 상태 관리는 Zustand 또는 Context API
- **백엔드**: Node.js + Express + `socket.io`
- **데이터**: 단어 목록은 JSON 파일로 관리, 방/플레이어 상태는 서버 인메모리(Map)로 관리 (필요 시 추후 Redis로 확장)
- **배포**: 무료/저비용 옵션 고려 (예: 프론트는 Vercel/Netlify, 백엔드(소켓 서버)는 Render/Railway 등 상시 구동 가능한 서비스)

## 12. 반드시 처리해야 할 예외 상황

- 닉네임 중복 입력 시 안내 및 재입력 요구
- 방 인원이 30명 초과 시 입장 거부 및 안내 메시지
- 존재하지 않는 방 코드 입력 시 에러 안내
- 게임 진행 중 참가자가 이탈(연결 끊김)해도 나머지 인원의 진행에 영향 없도록 처리 (전원 제출 판정 시 이탈한 인원은 제외하고 계산)
- 모바일/태블릿 화면 크기에서도 알파벳 버튼과 텍스트가 잘 보이도록 반응형 레이아웃 적용

## 13. 산출물 요청 형식

다음을 모두 포함해서 만들어줘:

1. 전체 폴더 구조 (frontend / backend 분리)
2. 백엔드: Express + Socket.io 서버 코드 (방 생성/참가/게임 진행/자동 정리 로직 포함)
3. 프론트엔드: React 컴포넌트 (로비, 대기실, 게임 화면, 랭킹 화면)
4. 샘플 단어 데이터 JSON (난이도별로 최소 10개씩)
5. 실행 방법을 안내하는 README.md (설치, 로컬 실행, 환경변수 등)

위 사양을 기준으로 실제 동작하는 최소 기능 제품(MVP)을 먼저 완성하고, 이후 확장 가능한 구조로 코드를 작성해줘.
